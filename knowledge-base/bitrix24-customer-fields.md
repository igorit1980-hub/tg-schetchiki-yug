# Bitrix24 Customer Fields

## 1. Цель

Это ТЗ фиксирует, какие поля должны быть у `Контактов` и `Компаний` в `Битрикс24` для работы Telegram-карты клиента, QR-кода, номера карты и допуска к оптовым ценам.

## 2. Базовая модель

- `Контакт` = основной клиентский объект
- `Компания` = контекст B2B-клиента
- карта клиента принадлежит `Контакту`
- QR-код и номер карты привязаны к `Контакту`

## 3. Поля Контакта

### Обязательные системные поля

- `NAME`
- `LAST_NAME`
- `PHONE`
- `CITY` или пользовательское поле города
- `COMPANY_ID` при наличии компании
- `ASSIGNED_BY_ID`

### Обязательные пользовательские поля Telegram/loyalty

1. `UF_CRM_TG_USER_ID`
   Тип: строка
   Назначение: `telegram_user_id`

2. `UF_CRM_TG_USERNAME`
   Тип: строка
   Назначение: `telegram_username`

3. `UF_CRM_TG_SOURCE`
   Тип: список или строка
   Значение по умолчанию: `telegram`

4. `UF_CRM_PHONE_NORMALIZED`
   Тип: строка
   Назначение: нормализованный телефон для дедупликации

5. `UF_CRM_CUSTOMER_TYPE`
   Тип: список
   Значения:
   - `developer`
   - `wholesaler`
   - `installer`
   - `retail`
   - `other`

6. `UF_CRM_CLIENT_CARD_ID`
   Тип: строка
   Назначение: человекочитаемый номер карты
   Пример: `SY-104582`

7. `UF_CRM_CLIENT_QR_PAYLOAD`
   Тип: строка
   Назначение: payload для QR
   Пример: `LOYALTY:SY-104582`

8. `UF_CRM_CARD_STATUS`
   Тип: список
   Значения:
   - `not_created`
   - `issued`
   - `active`
   - `blocked`
   - `archived`

9. `UF_CRM_APPROVAL_STATUS`
   Тип: список
   Значения:
   - `new`
   - `pending_review`
   - `approved`
   - `rejected`

10. `UF_CRM_ALLOWED_PRICE_TYPE`
    Тип: список
    Значения:
    - `retail`
    - `wholesale`

11. `UF_CRM_DISCOUNT_PERCENT`
    Тип: число
    Назначение: размер скидки или бонусного допуска

12. `UF_CRM_APPROVED_AT`
    Тип: дата/время

13. `UF_CRM_REJECTED_AT`
    Тип: дата/время

14. `UF_CRM_LAST_SYNC_AT`
    Тип: дата/время

15. `UF_CRM_COMPANY_NAME_SNAPSHOT`
    Тип: строка
    Назначение: слепок названия компании на момент регистрации

16. `UF_CRM_CARD_COMMENT`
    Тип: текст
    Назначение: комментарий менеджера по карте

## 4. Поля Компании

Минимально нужны:

1. `TITLE`
2. `PHONE`
3. `EMAIL`
4. `ASSIGNED_BY_ID`

### Рекомендуемые пользовательские поля Компании

1. `UF_CRM_COMPANY_TYPE_B2B`
   Тип: список
   Значения:
   - `developer`
   - `wholesaler`
   - `installer`
   - `dealer`
   - `other`

2. `UF_CRM_COMPANY_INN`
   Тип: строка

3. `UF_CRM_COMPANY_SYNC_SOURCE`
   Тип: строка
   Значение: `telegram/site/manual`

## 5. Правила заполнения

### При создании клиента из Telegram

Обязательно заполнить у контакта:

- имя
- телефон
- нормализованный телефон
- город
- тип клиента
- `UF_CRM_TG_USER_ID`
- `UF_CRM_TG_USERNAME` если есть
- `UF_CRM_TG_SOURCE = telegram`
- `UF_CRM_APPROVAL_STATUS = pending_review`
- `UF_CRM_ALLOWED_PRICE_TYPE = retail`
- `UF_CRM_CARD_STATUS = not_created` или `issued`

### При одобрении клиента менеджером

Нужно заполнить или обновить:

- `UF_CRM_CLIENT_CARD_ID`
- `UF_CRM_CLIENT_QR_PAYLOAD`
- `UF_CRM_CARD_STATUS = active`
- `UF_CRM_APPROVAL_STATUS = approved`
- `UF_CRM_ALLOWED_PRICE_TYPE = wholesale`
- `UF_CRM_DISCOUNT_PERCENT`
- `UF_CRM_APPROVED_AT`

### При отказе

Нужно заполнить или обновить:

- `UF_CRM_APPROVAL_STATUS = rejected`
- `UF_CRM_ALLOWED_PRICE_TYPE = retail`
- `UF_CRM_REJECTED_AT`

## 6. Правила привязки к Компании

- если клиент B2B, контакт должен быть привязан к компании
- если компании нет, менеджер создает ее вручную или через отдельный процесс
- если клиент розничный, компания не обязательна

Правило:

- контакт не должен автоматически клонироваться под каждую компанию
- один человек должен оставаться одним контактом

## 7. Правила номера карты и QR

Номер карты:

- хранится в `UF_CRM_CLIENT_CARD_ID`
- должен быть уникальным
- должен быть пригоден для ручного ввода

QR payload:

- хранится в `UF_CRM_CLIENT_QR_PAYLOAD`
- должен быть уникальным
- должен ссылаться на номер карты или защищенный loyalty token

В Telegram-карте должны отображаться:

- ФИО
- номер карты
- QR-код
- статус карты
- разрешенный тип цены

## 8. Логика поиска клиента

Порядок поиска:

1. по `UF_CRM_PHONE_NORMALIZED`
2. если не найден, по `UF_CRM_TG_USER_ID`
3. если найдено несколько контактов, нужен ручной разбор менеджером

## 9. Правило допуска к оптовому режиму

Контакт получает оптовый режим только если:

- `UF_CRM_APPROVAL_STATUS = approved`
- `UF_CRM_CARD_STATUS = active`
- `UF_CRM_ALLOWED_PRICE_TYPE = wholesale`

Если хотя бы одно условие не выполнено:

- Telegram показывает только розничные цены

## 10. Что должен видеть менеджер

Менеджеру должно быть достаточно карточки контакта, чтобы понять:

- кто это
- из какого источника пришел клиент
- к какой компании он относится
- одобрена ли карта
- какой номер карты выдан
- какой QR связан с картой
- разрешен ли оптовый режим

## 11. Практическая рекомендация

Для первой версии лучше:

- не делать отдельный смарт-процесс под карту клиента
- вести карту как атрибут `Контакта`
- вести B2B-контекст через `Компанию`

Так проще синхронизировать Telegram, магазин и CRM без лишних сущностей.
