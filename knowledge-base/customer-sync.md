# Customer Sync Logic

## 1. Цель

Нужна единая логика для связки:

- Telegram-канал / будущий Telegram-бот
- карта постоянного клиента
- `Битрикс24`

Эта логика определяет:

- как клиент появляется в системе
- как он ищется и обновляется
- кто является мастер-источником статуса клиента
- когда Telegram должен показывать розничную или оптовую цену

## 2. Главный принцип

- Telegram является точкой входа клиента
- `Битрикс24` является мастер-источником карточки клиента и статуса допуска
- сайт и Telegram не принимают окончательное решение по оптовому статусу без подтверждения из `Битрикс24`

## 3. Основные состояния клиента

Нормализованные состояния:

1. `guest`
   Клиент пришел из канала, не зарегистрирован

2. `registration_submitted`
   Клиент отправил заявку на карту, запись создана или обновлена в `Битрикс24`

3. `pending_review`
   Заявка существует, менеджер еще не подтвердил тип клиента

4. `approved_wholesale`
   Менеджер подтвердил клиента как застройщика, оптовика, монтажника или другого допустимого B2B-клиента

5. `rejected`
   Менеджер отказал в оптовом статусе

6. `archived`
   Карта или заявка деактивирована, клиент больше не должен получать оптовый режим

## 4. Что видит клиент в Telegram

### `guest`

- видит только розничные цены
- может смотреть каталог
- видит CTA на оформление карты клиента

### `registration_submitted`

- видит только розничные цены
- видит статус: заявка отправлена
- получает экран ожидания проверки

### `pending_review`

- видит только розничные цены
- видит, что карта оформляется и ожидает подтверждения менеджера

### `approved_wholesale`

- видит только оптовые цены
- получает активную карту клиента
- может пользоваться оптовым кабинетом

### `rejected`

- видит только розничные цены
- видит отказ или необходимость уточнить данные

### `archived`

- видит только розничные цены
- карта считается недействительной

## 5. Мастер-источники по данным клиента

### Telegram

Хранит только операционные данные сессии:

- `telegram_user_id`
- `telegram_username`
- `telegram_chat_id`
- локальный статус интерфейса
- время последней синхронизации

### Bitrix24

Хранит каноническое состояние клиента:

- карточка клиента
- статус заявки
- тип клиента
- одобрен ли оптовый доступ
- id карты
- размер скидки
- разрешенный тип цены
- ответственный менеджер

## 6. Канонические ключи поиска клиента

Основной ключ:

- телефон в нормализованном формате

Дополнительный ключ:

- `telegram_user_id`

Дополнительные вспомогательные поля:

- `telegram_username`
- `client_card_id`
- внутренний `Bitrix24` id контакта или смарт-процесса

## 7. Правило дедупликации клиента

### Основное правило

В системе должен существовать один активный клиент на один нормализованный телефон.

### Правило поиска

1. сначала искать по телефону
2. если телефон не найден, искать по `telegram_user_id`
3. если найдено несколько активных записей, отдавать в ручную проверку менеджеру

### Правило записи

- если найден существующий клиент, делать update
- если клиент не найден, делать create
- если найден конфликт по телефону и `telegram_user_id`, помечать запись как конфликтную

## 8. Какие данные клиент передает при регистрации

Минимальный обязательный набор:

- имя
- телефон
- город
- тип клиента

Рекомендуемый расширенный набор:

- фамилия
- компания
- ИНН
- комментарий
- источник регистрации = Telegram

## 9. Какие поля нужны в Bitrix24

Целевая модель проекта:

- клиент хранится в `Контактах`
- принадлежность клиента к юрлицу хранится через `Компании`
- карта клиента и Telegram-признаки хранятся как поля контакта

Отдельный смарт-процесс для клиента в первой версии не обязателен.

Минимальный набор полей:

- `full_name`
- `phone_normalized`
- `city`
- `customer_type`
- `telegram_user_id`
- `telegram_username`
- `telegram_channel_source`
- `client_card_id`
- `client_qr_payload`
- `card_status`
- `approval_status`
- `allowed_price_type`
- `discount_percent`
- `assigned_manager_id`
- `approved_at`
- `rejected_at`
- `last_sync_at`
- `company_id`
- `company_name_snapshot`

## 9.1. Практика учета через Контакты и Компании

Рекомендуемая практика:

- один физический человек = один `Контакт`
- один контрагент = одна `Компания`
- контакт может быть привязан к одной основной компании
- если человек работает с несколькими компаниями, это должно отдельно проверяться менеджером, а не заводиться автоматически дублями

Карта клиента должна принадлежать контакту, а не компании.
Компания нужна для понимания, к какому B2B-контексту относится контакт:

- монтажная организация
- застройщик
- оптовая компания
- розничный клиент без компании

QR-код и номер карты должны ссылаться именно на контакт, потому что в Telegram взаимодействует человек, а не юрлицо.

## 10. Нормализованные значения

### `customer_type`

- `developer`
- `wholesaler`
- `installer`
- `retail`
- `other`

### `card_status`

- `not_created`
- `issued`
- `active`
- `blocked`
- `archived`

### `approval_status`

- `new`
- `pending_review`
- `approved`
- `rejected`

### `allowed_price_type`

- `retail`
- `wholesale`

## 10.1. Правило карты и QR-кода

Рекомендуемая модель:

- номер карты = уникальный идентификатор контакта в loyalty-логике
- QR-код содержит только безопасный идентификатор карты или контакта
- QR-код не должен содержать открыто телефон, ФИО или другие персональные данные

Практический вариант:

- `client_card_id` = внутренний номер карты
- `client_qr_payload` = токен или код вида `LOYALTY:104582`

В Telegram-карте должны одновременно показываться:

- QR-код
- номер карты для ручного ввода

Рекомендуемый формат отображаемого номера карты:

- `SY-104582`
- или `UG-104582`

При сканировании в магазине или в системе:

1. ищется контакт по `client_card_id` или `client_qr_payload`
2. определяется контакт
3. определяется привязанная компания
4. определяется разрешенный тип цены
5. определяется скидка и статус карты

## 11. Правила синхронизации Telegram -> Bitrix24

### Событие: клиент заполнил форму на карту

Нужно:

1. нормализовать телефон
2. собрать Telegram-идентификаторы
3. найти клиента по телефону
4. если не найден, искать по `telegram_user_id`
5. создать или обновить запись в `Битрикс24`
6. зафиксировать источник регистрации = Telegram
7. установить статусы:
   - `approval_status = pending_review`
   - `allowed_price_type = retail`
8. вернуть в Telegram состояние `pending_review`

## 12. Правила синхронизации Bitrix24 -> Telegram

### Событие: менеджер одобрил клиента

Если в `Битрикс24`:

- `approval_status = approved`
- `card_status = active`
- `allowed_price_type = wholesale`

тогда Telegram должен:

- показать активную карту клиента
- переключить интерфейс на оптовые цены
- открыть доступ к оптовому кабинету

### Событие: менеджер отклонил клиента

Если в `Битрикс24`:

- `approval_status = rejected`

тогда Telegram должен:

- скрыть оптовый режим
- оставить только розничные цены
- показать статус отказа или предложение связаться с менеджером

### Событие: карта деактивирована

Если в `Битрикс24`:

- `card_status = blocked`
или
- `card_status = archived`

тогда Telegram должен:

- отключить оптовый режим
- показывать только розничные цены

## 13. Критерии получения оптового режима

Клиент получает оптовый режим только если одновременно:

- клиент существует в `Битрикс24`
- телефон клиента подтвержден или сохранен корректно
- `approval_status = approved`
- `card_status = active`
- `allowed_price_type = wholesale`

Если хотя бы одно условие не выполнено:

- Telegram показывает только розничный режим

Дополнительное правило:

- если контакт привязан к компании, компания может использоваться как подтверждающий B2B-признак
- но сама по себе привязка к компании не должна автоматически открывать оптовые цены без решения менеджера

## 14. Сценарии движения клиента

### Сценарий A. Новый клиент из канала

1. клиент приходит в Telegram-канал
2. смотрит каталог по розничным ценам
3. нажимает оформление карты
4. заполняет форму
5. запись уходит в `Битрикс24`
6. клиент получает статус `pending_review`

### Сценарий B. Менеджер подтверждает клиента

1. менеджер открывает запись в `Битрикс24`
2. проверяет клиента
3. назначает тип клиента
4. активирует карту
5. переводит клиента в оптовый режим
6. Telegram на следующей синхронизации открывает оптовые цены

### Сценарий C. Повторный вход клиента

1. клиент возвращается в Telegram
2. Telegram ищет текущий статус клиента
3. если карта активна, клиент сразу получает оптовый интерфейс
4. если карта не активна, клиент остается в розничном интерфейсе

## 15. Частота синхронизации

### Telegram -> Bitrix24

- мгновенная запись при отправке формы

### Bitrix24 -> Telegram

- быстрый pull статуса при открытии карты клиента
- плановая фоновая синхронизация каждые 5-15 минут

## 16. Что должен делать менеджер

Менеджер в `Битрикс24` должен:

1. открыть заявку клиента
2. проверить телефон и тип клиента
3. определить категорию клиента:
   - застройщик
   - оптовик
   - монтажник
   - розничный
4. при необходимости привязать контакт к компании
5. проверить, нет ли уже существующего контакта с тем же телефоном или Telegram
6. присвоить или подтвердить номер карты
4. принять решение:
   - одобрить
   - отклонить
7. при одобрении:
   - активировать карту
   - включить `allowed_price_type = wholesale`
8. при отклонении:
   - оставить `allowed_price_type = retail`

## 17. Правила безопасности и качества данных

- телефон должен храниться в нормализованном формате
- нельзя выдавать оптовый режим только по факту заполнения формы
- один клиент не должен иметь несколько активных карт одновременно
- Telegram не должен хранить окончательное решение по оптовому статусу локально без сверки с `Битрикс24`

## 18. Что нужно для следующего этапа разработки

Нужно определить:

- точную сущность в `Битрикс24`, где будут жить Telegram-клиенты
- точный набор пользовательских полей
- endpoint или метод backend-сервиса для:
  - создания/обновления клиента
  - получения статуса клиента по телефону или `telegram_user_id`
  - получения состояния карты клиента

## 19. Итоговое правило

Главное решение:

- Telegram приводит клиента
- `Битрикс24` принимает решение
- контакт является главным клиентским объектом
- компания является B2B-контекстом контакта
- Telegram показывает режим цен и карту только на основании подтвержденного статуса из `Битрикс24`
